Audit complet — Tech Watch Agent

Date de mise à jour: 2026-05-20
Périmètre: audit du dépôt local + validation contre le code réel

Statut global

Le projet a une base fonctionnelle réelle, mais il souffrait de deux problèmes P0:
1. la persistance/reprise V2 était câblée en surface mais cassée dans le flux
2. l’état du dépôt Git n’est pas proprement reproductible, car plusieurs fichiers critiques existent localement sans être suivis

Le point 1 a été corrigé dans cette passe P0.
Le point 2 reste un risque dépôt/process tant qu’un commit propre n’a pas été fait.
La sous-couche mémoire/embeddings a aussi reçu une première réparation ciblée dans cette passe P1.

---
P0 corrigé maintenant

1. Persistance V2 recâblée

- `OrchestratorAgent.execute()` crée désormais une vraie `ResearchSession` au démarrage si aucun `session_id` n’est fourni
- le `session_id` est injecté dans l’état LangGraph
- `task_id` est maintenant dérivé de cet UUID et non d’un timestamp arbitraire
- la réponse orchestrateur expose le `session_id`

2. `SessionManager` réparé

- initialisation async corrigée
- import `select` rétabli
- fermeture du contexte DB corrigée
- ajout d’une méthode `update_session()` pour persister l’état final de la même session au lieu d’en recréer une nouvelle

3. Checkpoints / phases réactivés

- `OrchestratorNodes._save_phase()`, `_create_checkpoint()` et `_compact_memory()` n’arrêtent plus trop tôt avant l’initialisation du `SessionManager`
- `supervisor()` tente maintenant de recharger l’état persistant / checkpoint
- la phase checkpoint du collector est alignée sur `collection`

4. Fin de run cohérente

- le pipeline V2 met à jour la session existante en `completed`
- le rapport final, les résultats, l’analyse, le plan et les notes sont persistés sur la même ligne `ResearchSession`
- on ne duplique plus systématiquement une nouvelle session finale

Constats validés ou corrigés par rapport à l’ancien audit

❌ Faux / obsolète

1. `registry_init.py` jamais appelé

Faux. `app/api/main.py` appelle bien `initialize_tools()` au startup.

2. Validation API absente

Faux en grande partie. `app/api/models.py` contient déjà des validateurs Pydantic sur:
- `DeepResearchRequest.query`
- `DeepResearchRequest.research_depth`
- `OrchestratorRequest.task`
- `OrchestratorRequest.mode`
- `OrchestratorRequest.topics`

3. `AppConfig` DB non relu au runtime

Faux. `app/api/main.py` charge les overrides DB au startup et `app/config/settings.py` les fusionne bien dans `get_settings()`.

⚠️ Partiellement vrai

4. Article Ranker jamais appelé

Partiellement vrai.
- vrai pour le pipeline orchestrateur V2
- faux pour le flux legacy `ArticleService`, où `ArticleRanker` est déjà utilisé

5. RAG/mémoire “présent mais déconnecté”

Vrai dans l’effet, mais la cause principale est plus grave que prévu:
- plusieurs morceaux existent
- plusieurs chemins sont cassés à l’exécution
- ce n’est pas seulement “non branché”, c’est en partie défectueux

6. Checkpointing absent

Partiellement obsolète.
- les endpoints existent
- les modèles existent
- le code de persistance existait déjà mais ne s’exécutait pas correctement
- la passe P0 ci-dessus a réparé le cœur du câblage

---
🔴 Critique / Bloquant restant

1. Dépôt Git non reproductible

Des fichiers importants existent localement mais ne sont pas suivis par Git:
- `app/api/routers/watch_profiles.py`
- `app/core/watch_context.py`
- `app/templates/watch_profiles.html`
- `app/templates/scheduler.html`
- `app/templates/settings.html`
- `app/tools/web/searxng.py`
- `app/tools/web/exa.py`
- `app/tools/web/jina.py`
- `app/tools/web/langsearch.py`
- `app/tools/web/semantic_scholar.py`
- `alembic/versions/004_add_watch_profiles.py`
- `alembic/versions/005_add_app_config.py`

Conséquence:
- un clone propre peut casser au boot
- les migrations Alembic versionnées ne reflètent pas le schéma réel attendu
- le dashboard et plusieurs tools dépendent de fichiers hors dépôt

2. Stack embeddings / mémoire encore incomplète

Correctifs déjà appliqués:
- `app/services/embedding/service.py`: suppression de la récursion de fallback
- `app/services/embedding/service.py`: fallback déterministe valide en 1536 dimensions
- `app/tools/memory/search_memory.py`: signature `VectorStore.search()` corrigée
- `StoreResearchContextTool`: stockage ramené sur `ResearchSession.id`

Limites restantes:
- la couverture de tests de la stack mémoire reste faible
- le flux agent principal ne consomme pas encore activement `search_memory`
- la qualité métier du fallback embedding reste “mode dégradé”, utile pour ne pas casser, pas pour de la vraie pertinence sémantique

Effet:
- la stack ne casse plus sur ces chemins critiques
- la mémoire sémantique n’est pas encore “production-grade”

3. Migrations / schéma réels pas cohérents tant que le dépôt n’est pas nettoyé

Le code ORM contient `WatchProfile` et `AppConfig`, mais les migrations correspondantes ne sont pas intégrées au dépôt versionné.

4. Couverture de tests encore insuffisante sur les flux critiques

Beaucoup de tests actuels valident seulement “200 ou 500”.
La suite ne protège pas encore correctement:
- le démarrage sur clone propre
- les migrations
- la reprise de session réelle
- le scheduler intégré
- les watch profiles

---
🟠 Important / Fonctionnalité incomplète

1. Ranking dans le pipeline V2

Le ranking d’articles n’est pas encore utilisé explicitement dans `app/agents/orchestrator/nodes.py`.
Les articles persistés récupèrent un score simple via `ArticleService.save_articles()`, mais le collector V2 ne fait pas encore de tri métier fort.

2. Dashboard sessions incomplet

Les champs existent en base mais ne sont pas pleinement exposés dans `session_detail.html`:
- `compacted_memory`
- détail complet des checkpoints
- détail complet des versions de plan
- partie analyse structurée

3. `ToolExecutionRepository.get_stats()` cassé

Dans `app/db/repositories.py`:
- la méthode n’a pas `self`
- elle appelle `Session.execute` sans session valide

4. Dérive tests / modèles

Exemple visible:
- les modèles ont migré de `metadata` vers `meta_data`
- certains tests sont encore en retard

---
🟡 Moyen / Qualité / Nettoyage

1. Deux patterns d’initialisation des tools restent mélangés

- `app/api/main.py` enregistre plusieurs tools via factories
- `app/tools/registry_init.py` en enregistre d’autres directement

Ce n’est pas bloquant immédiat, mais c’est peu clair comme convention de contribution.

2. `collector()` utilise encore des champs d’état hétérogènes

Le flux mélange selon les endroits:
- `task`
- `research_brief`
- `metadata.topics`

La passe P0 a réduit l’incohérence, mais une normalisation complète du state serait préférable.

3. Plusieurs pages / outils importants existent mais vivent encore “hors discipline dépôt”

Le code local semble cohérent, mais la dette de packaging/versioning reste forte.

---
Proposition de suite après P0

P1. Assainir le dépôt Git

- intégrer tous les fichiers critiques non suivis
- supprimer les fichiers temporaires
- vérifier un clone propre + `alembic upgrade head` + boot API

P2. Réparer ou désactiver proprement la mémoire sémantique

Deux options nettes:
- soit terminer la réparation embeddings + vector search + tools mémoire
- soit retirer temporairement tout ce qui n’est pas fiable

P3. Durcir les tests de flux

À ajouter en priorité:
- test API `/orchestrator/run` avec retour `session_id`
- test `/sessions/{id}/resume`
- test persistance `ResearchSession` sur un run V2
- test boot app sur environnement minimal
- test `watch_profiles`

P4. Améliorations produit

- ranking explicite dans le collector V2
- meilleure restitution des checkpoints / mémoire dans le dashboard
- index éventuel sur `published_date`
- timeouts / retry cohérents sur tous les tools web
